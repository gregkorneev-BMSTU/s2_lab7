# Лабораторная работа №7

Модификация контейнера из семинара 5 с заменой внутренней структуры хранения данных при сохранении внешнего интерфейса.

## Основа проекта

Проект представляет собой банковскую систему. В предметной области сохранены:

- банк и отделения;
- клиенты;
- счета разных типов;
- банковская карта;
- переводы между счетами;
- история операций;
- демонстрация полиморфизма.

Финальная версия проекта собрана на основе финальной версии семинара 5. Для лабораторной 7 поверх нее модифицирован контейнер `MyContainer<T>`: прежний внешний интерфейс сохранен, но внутренняя структура хранения данных заменена.

## Что требовалось

По заданию лабораторной работы нужно было:

- взять контейнер из семинара 5;
- заменить его внутреннюю структуру хранения;
- не менять внешний интерфейс контейнера;
- встроить обновленный контейнер в реальную прикладную задачу;
- уметь объяснить преимущества и недостатки выбранной структуры по сравнению с исходной реализацией и другими вариантами.

## Что сделано в проекте

В семинаре 5 `MyContainer<T>` был реализован как обычный динамический массив. В этой лабораторной работе контейнер переделан на сегментированный динамический массив:

- элементы больше не лежат в одном сплошном буфере;
- данные размещаются в нескольких блоках фиксированного размера;
- при этом пользовательский код продолжает работать через тот же набор методов, что и в семинаре 5.

Сохраненный внешний интерфейс контейнера:

- конструкторы копирования и перемещения;
- операторы присваивания;
- `push_back`;
- `emplace_back`;
- `pop_back`;
- `removeAt`;
- `erase`;
- `clear`;
- `reserve`;
- `size`;
- `empty`;
- `operator[]`;
- `find`;
- `begin`, `end`, `cbegin`, `cend`;
- `swap`.

## Выбранная внутренняя структура

Выбран собственный вариант внутренней структуры: сегментированный динамический массив.

### Идея реализации

Контейнер хранит не один большой массив, а таблицу указателей на блоки. Каждый блок содержит фиксированное количество ячеек для элементов.

Для обращения к элементу по индексу вычисляются:

- номер блока;
- смещение внутри блока.

При росте контейнера добавляются новые блоки, поэтому внутреннее хранение действительно отличается от исходного контейнера семинара 5.

## Почему интерфейс считается сохраненным

Внешнее поведение контейнера осталось прежним:

- элементы добавляются через `push_back` и `emplace_back`;
- удаляются через `pop_back`, `removeAt` и `erase`;
- доступны по индексу через `operator[]`;
- контейнер можно обходить через `begin()` и `end()`;
- контейнер можно передавать в `std::find` и `std::sort`;
- прикладной код классов `Bank`, `Client`, `Account` не пришлось переписывать под новую структуру хранения.

Именно это и означает сохранение интерфейса при изменении внутренней реализации.

## Итераторы

В контейнере реализован собственный random access итератор:

- поддерживаются `operator*`, `operator->`, `++`, `--`;
- поддерживаются арифметика итераторов и разность итераторов;
- поддерживаются сравнения;
- поддерживается доступ по смещению;
- итератор работает со стандартными алгоритмами.

Важно: итератор является random access, но контейнер не является contiguous-контейнером, потому что данные физически расположены по блокам, а не одним непрерывным массивом.

Сравнение на равенство и неравенство корректно работает и для итераторов разных контейнеров. Порядковые сравнения и разность итераторов имеют смысл только для итераторов одного и того же контейнера, поэтому в реализации предусмотрена защитная проверка.

## Асимптотика основных операций

- доступ по индексу `operator[]` - `O(1)`;
- `push_back` - амортизированно `O(1)`;
- `emplace_back` - амортизированно `O(1)`;
- `pop_back` - `O(1)`;
- `find` - `O(n)`;
- `removeAt` - `O(n)`;
- `erase` - `O(n)`;
- `clear` - `O(n)`;
- `reserve` - `O(k)`, где `k` — количество новых блоков, которые нужно выделить;
- обход контейнера - `O(n)`;
- сортировка через `std::sort` - `O(n log n)`.

## Преимущества и недостатки выбранной структуры

### По сравнению с исходной реализацией на обычном массиве

Преимущества:

- память расширяется блоками;
- не требуется переносить все элементы в один новый большой непрерывный буфер;
- внешний интерфейс остается таким же, как у исходного контейнера.

Недостатки:

- элементы не лежат подряд в одном участке памяти;
- доступ по индексу внутри реализации чуть сложнее;
- код контейнера сложнее, чем у обычного динамического массива.

## Сравнение с другими вариантами из задания

По сравнению со связанным списком:

- сохраняется быстрый доступ по индексу;
- проще поддерживать совместимость со `std::sort`.

По сравнению с кучей:

- сохраняется обычная последовательная модель хранения элементов;
- контейнер остается универсальным, а не заточенным только под приоритетный доступ.

По сравнению с деревом поиска:

- не нужно навязывать упорядоченность всем данным;
- интерфейс контейнера остается ближе к исходному контейнеру семинара 5.

Почему этот вариант удобен для защиты:

- легко показать, что внутренняя структура действительно изменена;
- легко показать, что внешний интерфейс сохранен;
- структура достаточно нетривиальна, но все еще понятна для объяснения на лабораторной.

## Где используется контейнер

Контейнер встроен в предметную модель проекта:

- `Bank` хранит отделения в `MyContainer<Department>`;
- `Bank` хранит клиентов в `MyContainer<Client*>`;
- `Client` хранит счета клиента в `MyContainer<Account*>`;
- `Account` хранит историю операций в `MyContainer<Transaction>`;
- в демонстрации полиморфизма используется `MyContainer<std::unique_ptr<Account>>`.

Таким образом, контейнер используется внутри реальной прикладной логики, а не только в отдельном учебном примере.

## Что демонстрирует программа

`src/main.cpp` показывает:

- работу контейнера на пользовательских типах;
- добавление элементов;
- удаление элементов;
- поиск через `std::find`;
- сортировку через `std::sort`;
- обход элементов через итераторы и range-based `for`;
- работу контейнера с `std::unique_ptr`;
- сохранение всей банковской логики после замены внутренней структуры;
- отдельную демонстрацию того, что для лабораторной 7 интерфейс сохранен, а внутренняя структура изменена.
